What is cdc?

cdc (Chrome DevTools CLI) is a project-agnostic command-line tool for interacting with Chrome pages and extensions during development. It connects directly to Chrome via the Chrome DevTools Protocol (CDP) on port 9222, giving you fine-grained control without any project-specific code.

Key insight: cdc uses the Accessibility Tree (via CDP) to understand page structure — no vision model needed. Text models navigate 3× faster when fed structured element tables vs. screenshots.

Designed for:

  • Extension developers — reload extensions, read errors, toggle dev mode
  • AI agent tooling — programmatic fill, click, verify via numbered indices
  • Test automation — bulk form filling, structured layout extraction
  • Performance debugging — every operation reports its duration in ms

Installation

# Clone and setup
git clone https://github.com/shivam2014/chrome-cdp-cli.git
cd chrome-cdp-cli
npm install
npm link            # makes `cdc` available globally

# Chrome must be running with remote debugging enabled
/Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome \
  --remote-debugging-port=9222

# Verify it works
cdc status

Alternatively, set the CDC_PORT environment variable to use a different CDP port.

Layout Modes

cdc layout extracts page structure without a vision model — it uses the browser's accessibility tree. Two modes serve different workflows:

Actionable (default)

Numbered interactive elements as a markdown table. AI agents reference elements by number: "fill element 5". Fast, compact, directly actionable.

Fill Engine

The fill command uses an intelligent auto-detection engine that adapts to each element type. It resolves targets by index number (from layout) or CSS selector, then applies the optimal strategy.

Element Type Detection

The engine classifies every element before filling:

  • Text inputs — <input>, <textarea> → paste strategy with char-by-char fallback
  • Select dropdowns — <select> → three-phase matching (exact → startsWith → includes)
  • Checkboxes / Radios — toggles checked state automatically
  • File inputs — detected and rejected (unsupported)

Fill Strategies

For text inputs, the engine tries two strategies in order:

Paste (primary)

1. el.focus()
2. el.value = ''
3. el.value = value
4. dispatch InputEvent with
   inputType='insertFromPaste'
5. dispatch 'change' event
6. el.blur()
7. Verify el.value === value
   → strategy: 'paste'

Char-by-char (fallback)

1. el.focus()
2. el.value = ''
3. for each character:
   el.value = substring
   dispatch 'input' event
4. dispatch 'change' event
5. el.blur()
6. Verify el.value === value
   → strategy: 'char-by-char'

Structured Results

Every fill operation returns a structured JSON result with all the information you need:

{
  "ok":      true,
  "actual":  "Bhalla",
  "strategy": "paste",
  "error":   null,
  "message": "OK",
  "duration": 12
}

Select Matching (3 Phases)

When a <select> element is detected, the engine uses a three-phase matching algorithm:

🎯

Phase 1 — Exact

Case-insensitive exact match on option text or value

🔤

Phase 2 — startsWith

Option text or value starts with the input

🔍

Phase 3 — includes

Option text or value contains the input (broadest match)

The --wait Flag

Some elements don't exist in the DOM immediately — they appear after async operations, API calls, or transitions. The --wait flag tells cdc to poll for the element before acting. 

# Wait up to 5 seconds for element to appear, then fill
cdc fill oraclecloud 5 "Bhalla" --wait 5000

# Wait up to 3 seconds for a dynamically loaded button
cdc click oraclecloud 8 --wait 3000
How it works: cdc runs a busy-poll loop in-page (200ms intervals) calling resolveElement() until the element is found or the deadline expires. If the element never appears, the result includes element_not_found with the wait time.

Duration Tracking

Every cdc command reports its execution time in milliseconds. This is invaluable for performance debugging — you can identify slow pages, heavy CDP calls, or network latency in your automation pipeline.

  "strategy": "paste",

Architecture

cdc's architecture is designed for reliability and clean separation of concerns:

→
connect.mjs
Playwright CDP
→
circuit.mjs
Circuit Breaker
→
commands/*.mjs
One per command
→
page-scripts.js
Browser context

Component Breakdown

🔌

connect.mjs

Wraps Playwright's connectOverCDP(). Provides connect() for tab listing and findPage() for URL-pattern matching.

⚡

circuit.mjs

Circuit breaker state machine: CLOSED → OPEN (2 failures) → HALF_OPEN (auto-test after 15s) → CLOSED. State persists in /tmp/cdc-circuit.json.

📋

commands/*.mjs

One file per command. Each receives CDP_URL and command-specific args, uses findPage() or connect(), and returns structured output.

🌐

page-scripts.js

Functions evaluated in the browser context via page.evaluate(). Handles element resolution, mouse events, text input, and dropdown inspection.

🧠

interact.mjs

Higher-order interaction engine. Loads page-scripts.js, builds resolve scripts (waitResolveScript for polling), executes in-page, returns structured results.

Data Flow

1  bin/cdc.mjs  parses args, routes to command
2  Command calls findPage(cdpUrl, pattern)
3    connect.mjs → connectOverCDP → returns {browser, ctx, page}
4    circuit.mjs checks state (fail fast if OPEN)
5  Command executes via page.evaluate(page-scripts.js)
6    Element resolution: index ↔ CSS selector
7    Action: fill, click, hover, etc. (in browser context)
8  Structured result → stdout
9  browser.close() → CDP connection closed

How It Works

Shadow DOM Drilling

Chrome extension pages use Shadow DOM heavily (e.g., chrome://extensions). cdc navigates these by manually drilling through shadowRoot properties: 

extensions-manager
  └─ shadowRoot
     └─ extensions-item-list
        └─ shadowRoot
           └─ extensions-item
              └─ shadowRoot
                 ├─ #name (text content)
                 ├─ #extension-id (text content)
                 └─ #dev-reload-button (click target)

This allows reload-ext and ext-errors to work with any Chrome extension without any project-specific code.

Accessibility Tree

The layout command uses CDP's Accessibility.getFullAXTree to extract the full accessibility tree. This gives a complete, structured view of the page — including elements in shadow DOMs — without needing a vision model: 

// CDP call used by tree mode
const cdp = await ctx.newCDPSession(page);
const { nodes } = await cdp.send('Accessibility.getFullAXTree');
renderTree(nodes, maxDepth);

Real Mouse Events

Unlike simple .click() calls, cdc's click and hover commands dispatch a full sequence of DOM events that mimic real user interaction: 

// Click sequence (real mouse events)
pointerdown → mousedown → pointerup → mouseup → click

// Hover sequence
pointerover → mouseover → pointerenter → mouseenter →
pointermove → mousemove

This ensures that JavaScript frameworks listening for specific event types (e.g., React's onClick via PointerEvent) respond correctly.

Element Resolution

Elements can be targeted by index number or CSS selector: 

// Index-based resolution (from layout output)
function resolveElement(target) {
  const isIndex = /^\d+$/.test(String(target));
  if (isIndex) {
    // Collect all interactive elements, sorted by Y then X
    // Filter out invisible/non-interactive elements
    return items[parseInt(target) - 1].el;
  }
  return document.querySelector(target);
}
